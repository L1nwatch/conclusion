# 决策模型设计

Conclusion 的决策工作台用于在形成最终结论前，强迫自己从多个角度检查问题。它保存的是推演过程，不代替最终的一句话结论。

## 设计原则

- 模型是可选工具，不是所有决定都必须完成的审批流。
- 每个模型使用稳定 ID 和固定字段，便于网页、搜索和 MCP 读取。
- 保存模型版本，未来调整问题时仍能解释旧记录。
- 原始回答保持简短纯文本；最终核心理由仍可使用 Markdown 附参考链接和公网图片。
- 不把第三方作者的思想包装成逐字、官方或唯一的检查清单。

## MVP 模型

### `time-horizons`

问题：这个决定在 10 小时、10 天、10 个月和 10 年后分别意味着什么？

该模型受 Suzy Welch 的 10-10-10 长短期视角启发，但时间单位是 Conclusion 针对个人工作流做的自定义版本，不宣称等同于原方法。

参考：[Suzy Welch 对 10-10-10 的介绍](https://suzywelch.com/books/)

### `scenario-range`

问题：合理范围内的最好、最可能和最坏结果是什么？如果最坏情况发生，如何降低损失或保留退路？

这里的重点不是预测极端事件，而是让上行空间、基准预期和可承受下行同时可见。

### `munger-checklist`

问题覆盖：激励、机会成本、反演、二阶效应、能力圈和反方证据。

名称使用“芒格式多模型检查”，而不是“查理·芒格官方 checklist”。Munger 强调需要把来自多个学科的模型组成 latticework，并警告只使用一两个模型会让人扭曲现实以适配工具；本模块据此设计了一组适用于个人决策的提示词。

参考：[Charlie Munger 1994 年 USC 演讲整理](https://www.ivey.uwo.ca/media/2975916/the-best-of-charlie-munger-1994-2011.pdf)

## 后续扩展

新模型需要：

1. 新增稳定且不可复用的 `modelId`。
2. 在 API 中增加明确的字段 schema 和长度限制。
3. 在前端注册名称、说明和提示词。
4. 用 API、迁移、创建、编辑和详情测试固定行为。
5. 如果修改已有字段含义，增加顶层 schema 版本，不静默解释旧数据。

候选模型包括可逆性/单向门、决策矩阵、预演失败（pre-mortem）和概率校准。MVP 不引入自由拖拽节点、复杂分支图或自动 AI 结论。
